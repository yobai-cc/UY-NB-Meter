# UY-NB-Meter 部署文档

## 1. 文档目标

本文档用于说明 `UY-NB-Meter` 项目的部署方式，适用于当前这套单文件 Flask 服务。

当前已知线上域名为：

- `https://hmwssbapi.bovetech.cn:15001/`

当前代码默认本机监听：

- `127.0.0.1:15555`

推荐部署方式为：

1. Flask 服务只监听本机 `127.0.0.1:15555`
2. 由反向代理统一对外暴露 HTTP/HTTPS
3. 通过域名 `hmwssbapi.bovetech.cn` 对外提供访问

## 2. 部署架构

建议采用如下结构：

```text
设备 / 浏览器
        |
        v
反向代理（Caddy）
        |
        v
127.0.0.1:15555
        |
        v
Flask server.py
```

说明：

- Flask 本身负责接口处理和页面展示
- 反向代理负责域名绑定、HTTPS 证书、端口暴露和转发
- Flask 通过请求头 `X-Forwarded-Proto` 判断当前访问是 HTTP 还是 HTTPS

## 4. 启动方式

### 4.1 直接启动

在项目目录下执行：

```powershell
cd D:\apps\UY-NB-Meter
.\venv\Scripts\python.exe .\server.py
```

启动后服务监听：

- `127.0.0.1:15555`

可本机访问：

- `http://127.0.0.1:15555/`
- `http://127.0.0.1:15555/HMWSSBAPI/PostMeterReadingData`

### 4.2 当前代码默认配置

当前 `server.py` 末尾写死的启动参数为：

```python
app.run(debug=True, host='127.0.0.1', port=15555)
```

含义如下：

- 仅本机访问，外部无法直连 Flask 端口
- 使用 Flask 自带开发服务器
- `debug=True` 适合调试，不建议生产环境长期使用

## 5. 反向代理部署

### 5.1 推荐思路

建议保持与原项目一致的对外行为：

- 外部 HTTP：`http://hmwssbapi.bovetech.cn:15000`
- 外部 HTTPS：`https://hmwssbapi.bovetech.cn:15001`
- 内部转发：`http://127.0.0.1:15555`

历史备份代码中曾记录过一套旧部署方式：

- 内部端口：`15555`
- 外部 HTTP：`15000`
- 外部 HTTPS：`15001`
- 代理组件：`Caddy`

所以如果你们线上已经有 Caddy，也可以继续沿用。

### 5.2 Caddy 配置示例

```caddy
# ---------------------------------------------------------
# 1. HTTPS 通道 (端口 15001)
# 用途: Web 控制台 (安全登录) & 支持 SSL 的新水表
# ---------------------------------------------------------
https://hmwssbapi.bovetech.cn:15001 {
        # 代理给后端的 Python 程序
        reverse_proxy 127.0.0.1:15555
}

# ---------------------------------------------------------
# 2. HTTP 通道 (端口 15000)
# 用途: 旧水表上报 & 自动跳转
# ---------------------------------------------------------
http://hmwssbapi.bovetech.cn:15000 {
    # 逻辑 A: 精确匹配 API 路径 -> 必须反向代理，禁止跳转
    handle /HMWSSBAPI* {
        reverse_proxy 127.0.0.1:15555
    }

    # 逻辑 B: 匹配所有剩余路径 (如 / 或 /admin) -> 强制跳转 HTTPS
    handle {
        redir https://hmwssbapi.bovetech.cn:15001{uri}
    }
}
```

说明：

- `header_up X-Forwarded-Proto` 很重要
- 当前前端页面是否显示为 HTTP 或 HTTPS，依赖这个头

## 7. 部署步骤建议

建议按以下顺序执行：

1. 将项目拷贝到正式目录，例如 `D:\apps\UY-NB-Meter`

4. 本机启动 `server.py`

7. 配置域名解析和防火墙
8. 访问 `https://hmwssbapi.bovetech.cn:15001/` 验证页面
9. 使用 `curl` 验证上报接口

建议更新时按以下顺序：

1. 备份当前 `server.py`
2. 替换项目文件
3. 如依赖有变化，重新执行 `pip install -r requirements.txt`
4. 重启 Flask 服务
5. 重新验证首页和上报接口

## 9. Windows 服务托管建议

当前项目是直接运行 Python 脚本的方式。正式环境建议至少采用以下其中一种方式托管：

- Windows 任务计划程序开机启动
- NSSM 包装成 Windows 服务
- 其他进程守护工具

目标是避免以下问题：

- 用户退出终端后服务中断
- 服务器重启后服务未自动恢复
- 服务异常退出无人拉起

## 10. 生产环境建议

当前代码是演示/联调友好型实现，生产部署前建议至少关注以下几点：

### 10.1 关闭 Debug

当前代码中为：

```python
app.run(debug=True, host='127.0.0.1', port=15555)
```

生产环境建议改为：

```python
app.run(debug=False, host='127.0.0.1', port=15555)
```

### 10.2 加强鉴权

当前逻辑是：

- 带了 `Authorization` 但不匹配时，返回 `401`
- 不带 `Authorization` 时，当前代码仍会放行

如果生产环境需要严格控制，建议改成：

- 必须带 `Authorization`
- 且必须命中白名单

### 10.3 增加日志

当前项目主要依赖页面中的内存历史记录排查问题，不适合作为长期日志方案。建议增加：

- 请求日志
- 错误日志
- 服务启动日志

### 10.4 增加持久化

当前 `HISTORY_DATA` 只保存在内存中，服务重启后历史数据会丢失。若需要留档，建议接入：

- SQLite
- MySQL

## 11. 常见问题排查

### 11.1 页面打不开

检查项：

- Flask 是否启动成功
- `127.0.0.1:15555` 是否能本机访问
- 反向代理是否已启动
- 域名解析是否正确
- 防火墙是否放行外部端口

### 11.2 页面显示为 HTTP 而不是 HTTPS

检查反向代理是否正确传递：

- `X-Forwarded-Proto: https`

因为当前代码使用这个请求头判断安全协议。

### 11.3 接口返回 401

检查项：

- `Authorization` 是否正确
- 是否使用了白名单中的 Basic 值

当前白名单为：

- `Basic ZWRwOk5hdmF5dWdhMTIz`
- `Basic YWRtaW46MTIzNDU2`

### 11.4 接口返回 400

重点检查：

- 请求体是否为空
- 字段数是否为 14 或 15
- `CAN` 是否为 9 位数字
- `RSSI` 是否小于 12
- 流量、电压、告警字段是否为合法数值

### 11.5 数据重启后丢失

这是当前实现的正常现象，因为项目只使用内存缓存，没有数据库。

## 13. 部署结论

这个项目当前最适合的部署方式，是“Flask 本机监听 + Caddy 对外代理”的轻量模式。按原项目保留 `15000/15001` 双端口后，外部访问行为也能尽量保持一致。

如果后续准备长期生产运行，建议下一步优先补齐：

- 关闭 `debug=True`
- 加强鉴权
- 增加服务托管
- 增加日志
- 增加数据库持久化
